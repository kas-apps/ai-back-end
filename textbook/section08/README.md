# セクション8：設計と改善

## このセクションの目的

セクション7で、アドレス帳アプリが一通り動く状態になりました。  
しかし、「動く」と「良いコード」は別の話です。

このセクションでは、現在のコードを改めて見直し、設計上の問題点を整理します。  
大きなリファクタリングはせず、「次に何を直すべきか」が分かる状態を目指します。

## 現在のコードを振り返る

### api/index.php の問題点

セクション6で作った `api/index.php` を見直してみます。

現在のファイルは、次のすべての処理が1ファイルに詰まっています。

- DBへの接続
- URLとメソッドの解析（ルーティング）
- 一覧取得・1件取得・登録・更新・削除の5つの処理

処理が増えるほどファイルが長くなり、どこに何があるかが分かりにくくなります。  
今は1ファイルなので大きな問題には見えません。  
しかし、今後ファイルを分けていくと、DB接続のような共通処理を複数の場所に書いてしまう可能性があります。

### フロントエンドの問題点

`app.js` も同様に、すべての処理が1ファイルにまとまっています。  
現時点ではまだ許容範囲ですが、機能が増えると見通しが悪くなります。

加えて、現状のコードには**バリデーション（入力チェック）が不足しています**。  
名前が空のまま登録フォームを送信した場合、どうなるでしょうか？

## 責務分離とは何か

### 各層の役割を明確にする

「責務分離」とは、**それぞれのコードが担当する役割を明確に分けること**です。

このアプリの3層の役割をあらためて整理します。

| 層 | 担当する責務 |
|---|------------|
| フロントエンド | 画面の表示・ユーザーの入力受け取り・APIの呼び出し |
| バックエンド | データの検証・処理・DBへの命令 |
| データベース | データの永続的な保存 |

### フロントエンドとバックエンド、どちらでバリデーションするのか

ユーザーが入力したデータのチェックは、フロントエンドでもバックエンドでも行えます。  
それぞれの役割を整理します。

| 場所 | 役割 | 例 |
|------|------|---|
| フロントエンド | UXの改善（即時フィードバック） | `required` 属性・JavaScriptでのチェック |
| バックエンド | データの正確性を保証する | PHPでの必須チェック・形式チェック |

ブラウザのフォームでは `required` 属性があるため、通常は空のまま送信できません。  
しかし、APIはブラウザのフォーム以外からも呼び出せます。  
例えば、`curl` を使えば、フロントエンドのチェックを通らずに直接APIへリクエストを送れます。

試しに、`curl` で `name` を空にして送信してみます。

```bash
curl -X POST http://localhost:8000/api/contacts \
  -H "Content-Type: application/json" \
  -d '{"name":"","email":"test@example.com"}'

# 実行例 → {"id":"4","message":"Created"}
```

> **サーバーサイドのバリデーションが本番です。**  
> フロントエンドのバリデーションはあくまで補助です。  
> 最終的にバックエンドがデータの正確性を保証する責務を持ちます。

## バリデーションを追加する

### サーバーサイドで name の必須チェックを追加する

登録（POST）と更新（PUT）の処理に、`name` が空でないかを確認する処理を追加します。

```php
// 登録（POST）の処理に追加
if ($method === "POST") {
  $input = json_decode(file_get_contents("php://input"), true);

  if (empty($input["name"])) {
    http_response_code(400);
    echo json_encode(["error" => "name is required"]);
    exit;
  }

  // 以降は既存のINSERT処理...
}
```

```php
// 更新（PUT）の処理に追加
if ($method === "PUT" && $id !== null) {
  $input = json_decode(file_get_contents("php://input"), true);

  if (empty($input["name"])) {
    http_response_code(400);
    echo json_encode(["error" => "name is required"]);
    exit;
  }

  // 以降は既存のUPDATE処理...
}
```

### 確認する

curlでnameを空にして送信します。

```bash
curl -X POST http://localhost:8000/api/contacts \
  -H "Content-Type: application/json" \
  -d '{"name":"","email":"test@example.com"}'
```

```json
{"error":"name is required"}
```

HTTPステータスコードが `400` で返ってきます。  
フロントエンド側でこのエラーレスポンスを受け取って、ユーザーにメッセージを表示する処理を追加することもできます。

## DB接続を切り出す

### 現在の問題

現状では `api/index.php` の冒頭に DB 接続のコードが書かれています。  
ファイルが増えたとき、全ファイルに同じ接続コードを書くと修正が大変になります。

### db.php に切り出す

DB接続の処理を別ファイルに切り出します。

`api/db.php` を作成します。

```php
<?php
function get_pdo(): PDO {
  $pdo = new PDO('sqlite:../contacts.db');
  $pdo->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);
  return $pdo;
}
```

`api/index.php` の冒頭を書き換えます。

```php
<?php
header("Content-Type: application/json");

require_once __DIR__ . '/db.php';
$pdo = get_pdo();
```

動作を確認します。

```bash
curl http://localhost:8000/api/contacts
```

以前と同じように一覧が返ってくれば、DB接続の切り出しは成功です。  
リファクタリングでは、動作を変えずにコードの構造だけを改善できているかを確認します。

### 変更の意味

| 変更前 | 変更後 |
|-------|-------|
| `index.php` に直接 `new PDO(...)` | `db.php` の `get_pdo()` を呼ぶ |
| DBパスの変更は `index.php` を直接編集 | `db.php` だけを修正すればよい |

この変更は機能には影響しません。  
「DBの設定は `db.php` に書く」というルールができることで、コードの見通しが良くなります。

## API設計を振り返る

### 設計時の判断を確認する

セクション2で設計したAPIエンドポイントを振り返ります。

```text
GET    /api/contacts        → 一覧取得
GET    /api/contacts/{id}   → 1件取得
POST   /api/contacts        → 登録
PUT    /api/contacts/{id}   → 更新
DELETE /api/contacts/{id}   → 削除
```

この設計にはいくつかの判断が含まれています。

| 判断 | 理由 |
|------|------|
| URLにリソース名（contacts）を使う | 「何を操作するか」がURLで分かる |
| 操作の種類はHTTPメソッドで表す | URLを増やさずに操作を区別できる |
| IDはURLのパスに含める | `/contacts/1` と `/contacts/2` を区別できる |

### 設計がないと何が起きるか

設計のルールを決めずに作ると、次のようなAPIになることがあります。

```text
/getContacts
/addContact
/deleteContact?id=1
/updateContactById
```

動きはしますが、命名に一貫性がなく、他の人（や未来の自分）が使いにくくなります。  
RESTful API でよく使われる設計ルールは、こうした問題を防ぐための**共通の考え方**です。
