# セクション2：バックエンドの基礎概念

## このセクションの目的

セクション1では「なぜバックエンドが必要か」を理解しました。  
このセクションでは、バックエンドが実際にどのように動いているかを、**言語に依存しない概念**として理解します。

PHPのコードを書く前に、「仕組みとして何が起きているのか」を理解しておくことが重要です。  
この概念が頭に入っていると、コードを見たときに「これは何をしているのか」がわかるようになります。

## Webアプリの全体構造

### 3つの層

セクション1で登場した図をもう一度確認します。

```text
フロントエンド ──[リクエスト]─▶ バックエンド ──[操作]─▶ データベース
フロントエンド ◀─[レスポンス]── バックエンド ◀─[結果]── データベース
```

Webアプリは、大きく3つの層に分かれています。

| 層 | 別名 | 役割 |
|---|------|------|
| フロントエンド | クライアントサイド | ユーザーが操作する画面 |
| バックエンド | サーバーサイド | 処理を行う部分 |
| データベース（DB） | — | データを保存する部分 |

### なぜ層を分けるのか

層を分ける理由は「**役割を明確にするため**」です。

たとえば、アドレス帳に新しい連絡先を登録する場面を考えます。

- 登録する連絡先を入力する**画面を作る**のはフロントエンドの仕事
- 連絡先の**保存処理を実行する**のはバックエンド
- 連絡先を**保存する**のはデータベースの仕事

この分担が明確になっていると、問題が起きたときに「どの層が原因か」を判断できます。  
AIが生成したコードを読むときも、「このコードはどの層の話をしているのか」が分かれば、理解が格段に速くなります。

## フロントエンドとバックエンドはどう繋がるのか

### 問いかけ

> 「ボタンを押したとき、裏で何が起きているのか？」

フロントエンドとバックエンドは、別々のプログラムです。  
それぞれが独立して動いており、**HTTP**という通信の仕組みを使って会話しています。

### HTTPとは何か

**HTTP（HyperText Transfer Protocol）** は、Webブラウザとサーバーがデータをやりとりするためのルールです。

日常に例えると、HTTPは「電話のかけ方のルール」のようなものです。  
電話をかける側（フロントエンド）が、かけられる側（バックエンド）に向かって決まった形式で連絡します。

### クライアントとサーバー

| 呼び方 | 実体 | 役割 |
|--------|------|------|
| クライアント | ブラウザ・スマホアプリ | リクエストを送る側 |
| サーバー | バックエンドのプログラム | リクエストを受け取り、レスポンスを返す側 |

基本的なHTTP通信では、クライアントが先にリクエストを送ります。  
サーバーは、そのリクエストを受け取ってレスポンスを返します。

---

## なぜ「リクエスト」と「レスポンス」が必要なのか

### 問いかけ

> 「なぜサーバーは勝手にデータを送ってこないのか？」

これは、HTTPの基本的な設計によるものです。  
HTTPは「**一問一答**」の仕組みになっています。

```text
クライアント ──[リクエスト]─▶ サーバー
クライアント ◀─[レスポンス]── サーバー
```

1. クライアントが「リクエスト（要求）」を送る
2. サーバーが「レスポンス（応答）」を返す

通常のHTTP通信では、サーバーが何もない状態から勝手にレスポンスを送ることはありません。  
まずクライアントからリクエストがあり、それに対してサーバーがレスポンスを返します。

### なぜこの設計なのか

「クライアントが要求したときだけ、サーバーが答える」という設計は、シンプルで管理しやすいというメリットがあります。

サーバーが勝手に通知を送ってきたら、クライアント側は常に受け取る準備をしなければなりません。  
リクエスト/レスポンスの一方向モデルにすることで、処理の流れが追いやすくなっています。

### リクエストに含まれる情報

たとえば、アドレス帳に新しい連絡先を追加するとき、フロントエンドはバックエンドに対して次のようなリクエストを送ります。

```text
「/contacts に、新しい連絡先データを POST する」
```

これを分解すると、URL・メソッド・ボディという情報になります。

| 情報 | 役割 | 例 |
|------|------|---|
| URL | どこにアクセスするか | `/contacts` |
| メソッド | 何をしたいか | `GET`（取得）、`POST`（送信） |
| ボディ | 送りたいデータ | 登録する連絡先の情報 |

## なぜデータはJSONでやりとりするのか

### 問いかけ

> 「なぜHTMLじゃなくJSONなのか？」

フロントエンドとバックエンドがデータをやりとりするとき、多くの場合**JSON**という形式を使います。

### JSONとは何か

**JSON（JavaScript Object Notation）** は、データを文字列で表現するためのフォーマットです。

```json
{
  "id": 1,
  "name": "田中 太郎",
  "email": "taro@example.com",
  "phone": "090-1234-5678"
}
```

見た目はシンプルで、「キーと値のペア」でデータを表現します。

### なぜHTMLではないのか

HTMLは「画面を表示するためのもの」です。  
データのやりとりには余分な情報（タグなど）が多く含まれており、適していません。

JSONは「データだけ」を持ちます。シンプルで軽く、どのプログラミング言語でも扱えます。

HTMLを返す場合、サーバーは「完成した画面」を返します。  
JSONを返す場合、サーバーは「画面に表示するためのデータ」だけを返します。

バックエンドからデータだけ受け取りたい場合は、JSONの方が扱いやすくなります。

### JSONが言語非依存である理由

フロントエンドはJavaScript、バックエンドはPHP、場合によってはPythonやRubyを使うこともあります。  
JSONはテキストのフォーマットなので、**どの言語でも読み書きできます**。

これが、Webアプリのデータ交換にJSONが広く使われる理由です。

## APIとは何か、なぜ必要なのか

### 問いかけ

> 「なぜサーバーの処理を直接呼べないのか？」

バックエンドの処理は、外部から直接呼び出すことができません。  
**API（Application Programming Interface）** という「窓口」を通じてのみ、アクセスできます。

### APIを窓口に例える

APIは、お店の「注文カウンター」のようなものです。

- お客さん（クライアント）は、カウンター越しに注文する
- 厨房（バックエンドの処理）には直接入れない
- カウンター（API）が注文を受け取り、厨房に伝える

この仕組みにより、外部からは「何をお願いできるか」だけが見える状態になります。

### URLで機能を分ける

APIでは、URLごとに「何をするか」を対応させます。

APIは単なる入口ではなく、「どのURLに」「どのメソッドで」「どんなデータを送れば」「どんな結果が返るか」を決めた約束でもあります。

```text
GET    /contacts        → 連絡先の一覧を取得する
GET    /contacts/1      → id=1 の連絡先を取得する
POST   /contacts        → 新しい連絡先を登録する
PUT    /contacts/1      → id=1 の連絡先を更新する
DELETE /contacts/1      → id=1 の連絡先を削除する
```

このように、URLとHTTPメソッドを組み合わせてデータを操作するAPI設計は、**RESTful API**と呼びます。

### メソッドの意味

| メソッド | 意味 | 日本語のイメージ |
|---------|------|----------------|
| GET | 取得 | 見せて |
| POST | 作成 | 追加して |
| PUT | 更新 | 書き換えて |
| DELETE | 削除 | 消して |

---

### アドレス帳で一連の流れを見る

たとえば、新しい連絡先を登録するときは、次のような流れになります。

1. ユーザーがフロントエンドで名前・メールアドレス・電話番号を入力する
2. フロントエンドが `/contacts` に `POST` リクエストを送る
3. リクエストのボディには、JSON形式の連絡先データが入っている
4. バックエンドがデータを受け取る
5. バックエンドがDBに保存する
6. 保存結果をJSONでフロントエンドに返す
7. フロントエンドが画面を更新する

### セクション2のまとめ

| 概念 | 意味 |
|------|------|
| HTTP | フロントとバックエンドが会話するためのルール |
| クライアント / サーバー | 要求する側 / 応答する側 |
| リクエスト / レスポンス | 送る問い / 返ってくる答え |
| JSON | 言語を問わず使えるデータのフォーマット |
| API | バックエンドへのアクセス窓口（URL＋メソッド） |

これらの概念は、PHPでコードを書くときに繰り返し登場します。
