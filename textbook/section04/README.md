# セクション4：データが保存されない問題とデータ永続化

## このセクションの目的

セクション3では、PHPでJSONを返すAPIを作りました。  
しかし、そのAPIには根本的な問題があります。

**データが保存されない**、という問題です。

このセクションでは、その問題を正面から捉え直し、「なぜデータベースが必要なのか」を理解します。  
頭で理解するだけでなく、実際にデータが消える場面を体験することで、必要性を実感します。

このセクションでは、3つの保存方法を比べます。

1. PHPの変数：リクエスト中だけ使える一時的な置き場所
2. LocalStorage：ブラウザ内に残せる個人用の保存場所
3. データベース：Webアプリの正式な保存場所

この違いを理解すると、「なぜDBが必要なのか」が見えてきます。

---

## データが保存されない問題

### まず確認：固定データは「保存」ではない

`index.php`を、次のような処理に変更します。

```php
<?php
header("Content-Type: application/json");

// メモリ上に一時的にデータを保持する（例）
$contacts = [
  ["id" => 1, "name" => "田中 太郎", "email" => "taro@example.com"]
];

echo json_encode($contacts, JSON_UNESCAPED_UNICODE);
```

- `$contacts` に直接書いたデータは、毎回作られているだけ
- リロードしても残っているように見えるが、保存されているわけではない

### では、変数に追加すれば保存できるのか？

```php
<?php
header("Content-Type: application/json");

$contacts = [];

if ($_SERVER["REQUEST_METHOD"] === "POST") {
  $input = json_decode(file_get_contents("php://input"), true);

  $contacts[] = [
    "id" => 1,
    "name" => $input["name"],
    "email" => $input["email"]
  ];
}

echo json_encode($contacts, JSON_UNESCAPED_UNICODE);
```

このコードでは、POSTされたデータをいったん `$contacts` に追加し、その直後に `$contacts` をJSONとして返しています。
そのため、POSTした直後のレスポンスだけを見ると、登録できたように見えます。

- POSTで受け取ったデータを `$contacts[]` に追加する
- そのリクエスト中は使える
- でも次のリクエストでは `$contacts` が空に戻る

確認コマンド：

```bash
curl -X POST http://localhost:8000/api/index.php \
  -H "Content-Type: application/json" \
  -d '{"name":"田中 太郎","email":"taro@example.com"}'

# 実行結果 → [{"id":1,"name":"田中 太郎","email":"taro@example.com"}]
```

このときは一見、登録できたように見えます。
でも、次に GET しても空です。

```bash
curl http://localhost:8000/api/index.php

# 実行結果 → []
```

`[]` が返ってきます。  
つまり、さきほどPOSTした連絡先は残っていません。

なぜなら `$contacts = [];` がリクエストのたびに最初から実行されるからです。

### なぜ消えるのか

PHPの変数はメモリ上に存在します。  
メモリはプログラムが動いている間だけデータを保持し、プログラムが終了すると消えます。

```text
リクエストが来る
→ index.php が最初から実行される
→ 変数が作られる
→ レスポンスを返す
→ そのリクエスト中の変数は破棄される
```

PHPでは、リクエストごとに `index.php` が最初から実行されます。  
そのため、前のリクエスト中に作った変数の中身は、次のリクエストには引き継がれません。

> **「このままではアプリとして使えない」**  
> データが消えてしまうなら、どれだけ良いUIを作っても意味がありません。

## LocalStorageで解決できるのでは？

### ブラウザ上にデータを保存する方法

フロントエンドには、ブラウザ上にデータを保存できる **LocalStorage** という仕組みがあります。

```javascript
const contacts = [
  { name: "田中 太郎", email: "taro@example.com" }
];

// データを保存する
localStorage.setItem("contacts", JSON.stringify(contacts));

// データを取り出す
const savedContacts = JSON.parse(localStorage.getItem("contacts"));
console.log(savedContacts);
```

リロードしてもデータが残るため、「これで解決できるのでは？」と思うかもしれません。  
実際、ブラウザ上での簡単なメモや設定の保存には十分な場面もあります。

しかし、Web アプリに使おうとすると、すぐに限界が見えてきます。

---

### LocalStorageの限界

#### 問題1：ブラウザ・端末をまたいで共有できない

LocalStorageのデータは、そのブラウザの中に保存されます。

- 自分のPCのChromeに保存したデータは、スマホから見られない
- 同じPCでも、Chromeに保存したデータはFirefoxから見られない
- 他のユーザーと同じデータを共有できない

Web アプリとして、複数の端末・複数のユーザー間でデータを共有したい場合、LocalStorageにはその手段がありません。

> **「LocalStorageは"自分専用メモ帳"」**

#### 問題2：セキュリティが弱い

LocalStorageのデータは、ブラウザの開発者ツールから確認・変更できます。  
そのため、個人情報や認証情報のような重要なデータを安全に管理する場所としては向いていません。

#### 問題3：機能が弱い

LocalStorageはキーと値のペアを保存するシンプルな仕組みです。  
そのため、次のような操作が苦手です。

- 名前でデータを検索する
- 登録日時でデータを並び替える
- 複数の条件でデータを絞り込む

アドレス帳アプリでは当然必要になる操作ですが、LocalStorageだけでは実現が難しくなります。

---

## データベースとは何か

### 「サーバー側にデータを保存する」という発想

LocalStorageの限界を踏まえると、必要なものが見えてきます。

- **サーバー側**にデータを保存する
- 誰がアクセスしても、同じデータを参照できる
- 検索・並び替えなどの操作ができる

これを実現するのが **データベース（DB）** です。

### データの流れを整理する

データベースを使うと、データの流れは次のようになります。

```text
フロントエンド ──[リクエスト]─▶ バックエンド ──[操作]─▶ データベース
フロントエンド ◀─[レスポンス]── バックエンド ◀─[結果]── データベース
```

セクション1・2で繰り返し登場した図です。  
データベースは、この図の右端に位置しています。

バックエンドがデータベースにデータを書き込み、読み出す役割を担います。  
フロントエンドはデータベースに直接アクセスせず、必ずバックエンドを経由します。

> **「DBを使うにはバックエンドが必要」**  
> データベースへのアクセスはバックエンドが管理するため、フロントエンドだけでは完結しません。

### まとめ：LocalStorageとDBの違い

| 観点 | PHPの変数 | LocalStorage | データベース |
|------|-----------|--------------|--------------|
| 保存場所 | リクエスト中のメモリ | ブラウザ | サーバー側 |
| リロード後 | 残らない | 残る | 残る |
| データの共有 | できない | できない | できる |
| 検索・集計 | 不向き | 不向き | 得意 |
| 主な用途 | 一時的な処理 | 個人用の簡易保存 | アプリの正式な保存先 |

LocalStorageはブラウザ上の一時的な記憶領域として有用ですが、アプリのデータを本格的に管理するには適していません。

## この講座で扱うDB（SQLite）

### SQLiteの特徴

この講座では **SQLite** というデータベースを使います。

SQLiteの最大の特徴は、**1つのファイルがデータベースそのもの**という点です。

```text
contacts.db ← このファイルがデータベース
```

多くのデータベース（MySQLやPostgreSQLなど）は、別途サーバープロセスを起動する必要があります。  
SQLiteはファイルを作るだけで使い始められるため、環境構築がシンプルです。

### なぜこの講座でSQLiteを選ぶのか

| 理由 | 説明 |
|------|------|
| PHPから扱いやすい | 多くのPHP環境でSQLiteを利用できる |
| 追加サーバー不要 | MySQLのようなDBサーバーを別途起動しなくてよい |
| ファイル1つ | `contacts.db` というファイルがDBのすべて |
| 学習に最適 | 本番環境のDBと同じSQL構文が使える |

SQLiteは本番環境の大規模なアプリには向いていませんが、学習目的や小規模なアプリには十分です。  
また、MySQLなど他のデータベースに移行するときも、SQLの知識はそのまま活かせます。
