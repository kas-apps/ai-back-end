## 今のアプリの限界

### SQLiteの特徴と限界

今回使ったSQLiteは、ファイル1つで扱えるシンプルなデータベースです。  
学習やプロトタイプだけでなく、要件が合えば本番環境でも使われます。

ただし、どんな用途にも向いているわけではありません。

**同時書き込みには注意が必要**

SQLiteはファイルベースのデータベースです。  
読み取り中心の用途では十分に使えますが、多数のユーザーが同時に書き込みを行うサービスでは、書き込み待ちが発生しやすくなります。

**複数サーバー構成には向きにくい**

SQLiteは基本的に1つのDBファイルをアプリが直接扱います。  
複数のアプリサーバーから同じDBを安全に共有する構成は、MySQLやPostgreSQLのようなサーバー型DBの方が扱いやすくなります。

**DBサーバーとしての運用機能は少ない**

SQLiteはサーバー型DBではないため、DBユーザーごとの権限管理、ネットワーク越しの接続管理、レプリケーションなどはMySQLやPostgreSQLの方が得意です。

| 用途 | SQLite | MySQL / PostgreSQL |
|------|--------|--------------------|
| 学習・プロトタイプ | 向いている | セットアップが手間になることもある |
| 個人ツール・小規模アプリ | 向いている | 過剰になることもある |
| 読み取り中心の小規模本番アプリ | 要件が合えば使える | 使える |
| 高書き込み・高トラフィック | 向きにくい | 向いている |
| 複数サーバー構成 | 向きにくい | 向いている |

大切なのは、「SQLiteだから本番では使えない」と覚えることではありません。  
アプリの規模、同時書き込みの多さ、運用要件に合わせてDBを選ぶことです。

### PHPの直書きの限界

今回は `api/index.php` に処理を書きました。  
セクション8でDB接続を `db.php` に切り出しましたが、それでも以下のような限界があります。

- ルーティングを自前で書いている（`preg_match` での分岐）
- 認証・認可の仕組みがない
- コードが増えると管理が難しくなる

実際のサービスでは、これらをフレームワークが解決してくれます。

## 次のステップ

### MySQLへの移行

SQLiteからMySQLに移行するとき、PDOを使っていればPHP側の基本的な書き方は大きく変わりません。

```php
// SQLite
$pdo = new PDO('sqlite:../contacts.db');

// MySQL
$pdo = new PDO('mysql:host=localhost;dbname=myapp;charset=utf8', 'user', 'password');
```

PDO（PHP Data Objects）を使っているため、`prepare()` や `execute()` のような基本的な使い方は共通です。  
ただし、SQLの細かい構文、型、AUTO_INCREMENTの扱い、文字コード、運用方法には違いがあります。  
実際に移行するときは、DBごとの差分を確認する必要があります。

これが、セクション6でPDOを使った理由のひとつです。

### Laravelとは何か

**Laravel** は、PHPの代表的なWebフレームワークです。

今回自分で書いた処理の多くを、Laravelは仕組みとして提供しています。

| 今回自分で書いたこと | Laravelが提供するもの |
|--------------------|--------------------|
| `preg_match` でルーティング | `routes/web.php` でルート定義 |
| `json_decode(file_get_contents(...))` で入力取得 | `$request->input(...)` |
| `empty($name)` でバリデーション | `$request->validate(['name' => 'required'])` |
| CSRFトークンの実装 | ミドルウェアで自動対応 |
| PDOで直接SQL | Eloquent（ORM）でモデル経由 |

Laravelを使うと、ルーティング・バリデーション・認証・CSRF対策などが最初から組み込まれた状態で開発を始められます。

> **ただし：**  
> フレームワークを使っても、セキュリティの知識は必要です。  
> Laravelが提供する仕組みを「なぜ使うのか」を理解した上で使うことが重要です。  
> セクション9で学んだ知識は、Laravel上でも同じように役立ちます。

## AIとの向き合い方、改めて

セクション10でも触れましたが、このコースで学んだ内容はAIを活用するための基盤になります。

### 今できるようになったこと

- AIが生成したPHPのAPIを読んで、意図を理解できる
- プリペアドステートメントを使っているか確認できる
- `innerHTML` を使っていないか確認できる
- エラーメッセージから問題の場所を推測できる
- AIに具体的な条件を伝えて、より良いコードを引き出せる

### これから意識すること

AIは便利なツールです。しかし、生成されたコードを使うのは人間であり、動かす責任も人間にあります。

バックエンドの基礎を理解している人は、AIの出力を「評価」できます。  
これは、AIを道具として使いこなす上で、最も重要なスキルのひとつです。

> **「動くけど分からない」から「動くし分かる」へ。**  
> これがこのコースのゴールでした。
