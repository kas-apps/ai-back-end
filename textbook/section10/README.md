# セクション10：AIとどう向き合うか

## このセクションの目的

セクション9では、Webアプリのセキュリティについて学びました。  
SQLインジェクションやXSSがどのように起きるのか、そして今回のアプリでどのような書き方が対策になっているのかを確認しました。

このセクションでは、「AIとどう向き合うか」というテーマで、実務に使える考え方を整理します。  
AIを便利に使い続けるために、何を任せてよくて、何を自分で判断しなければならないのかを見ていきます。

---

## AIにコードを生成させてみる

### 実際にやってみる

セクション6で作ったのと同じAPIを、AIに生成させてみます。  
次のようなプロンプトを使ってみましょう。

```text
PHPとSQLiteを使って、連絡先を管理するAPIを作ってください。
GET/POST/PUT/DELETEに対応してください。
DB接続は api/db.php に分け、API本体は api/index.php に書いてください。
```

AIは動くコードを生成します。  
おそらく、`api/index.php` と `api/db.php` に近い構造のファイルが出力されます。

しかし、出力されたコードをそのまま使う前に、確認すべきことがあります。

### 生成されたコードを確認する

セクション9で学んだチェックポイントを使って確認します。

- **プリペアドステートメントを使っているか**
  - `$pdo->query("... WHERE name = '" . $name . "'")` のような書き方になっていないか
- **`innerHTML` を使っていないか**
  - フロントエンドも生成させた場合、表示処理に注意
- **バリデーションが入っているか**
  - `name` が空のまま登録できないか
- **エラー情報をそのまま返していないか**
  - `catch ($e) { echo $e->getMessage(); }` のような書き方は内部情報が漏れる
- **認証・認可が必要な操作を誰でも実行できないか**
  - 追加・更新・削除APIが無条件で呼び出せる状態になっていないか

AIが生成したコードは「動く」かもしれませんが、「安全」とは限りません。  
判断するのは、コードを読む人間です。

## プロンプトで結果は変わる

### 曖昧なプロンプトと具体的なプロンプト

同じ目的でも、プロンプトの書き方によってAIの出力は大きく変わります。

**曖昧なプロンプト：**

```text
PHPでAPIを作って
```

このプロンプトで生成されたコードは、次のような問題を含む可能性があります。

- セキュリティ対策がない
- バリデーションがない
- エラーハンドリングがない
- 用途に合わない構造になっている

**具体的なプロンプト：**

```text
以下の条件でPHPのAPIを作ってください。

- SQLiteを使う
- PDOとプリペアドステートメントを使う
- nameが空の場合は400エラーを返す
- エラーメッセージに内部情報を含めない
- api/index.php と api/db.php に分ける
- DB接続は api/db.php の get_pdo() にまとめる
```

条件を明示すると、AIはその条件に沿ったコードを生成しやすくなります。

> **ポイント：**  
> AIに「何を作るか」だけでなく「どう作るか」を伝えることが重要です。  
> セキュリティや設計の条件を明示しなければ、AIはデフォルトの（必ずしも安全ではない）書き方をします。

### プロンプトに含めると良いこと

| 項目 | 例 |
|------|---|
| 使用技術 | PHP / PDO / SQLite |
| セキュリティ要件 | プリペアドステートメントを使う |
| バリデーション要件 | name は必須、空なら400エラー |
| エラー処理の方針 | 内部情報をレスポンスに含めない |
| ファイル構成 | `api/index.php` と `api/db.php` に分ける |
| 認証・認可 | このサンプルでは未実装だが、本来は追加・更新・削除に必要と明記する |

これらを事前に伝えておくと、後から修正する手間が減ります。

### より良いプロンプト例

セクション8・9で学んだ内容まで反映するなら、次のように条件を増やします。

```text
PHPとSQLiteを使って、連絡先を管理するAPIを作ってください。

条件：
- GET /api/contacts で一覧取得
- POST /api/contacts で登録
- PUT /api/contacts/{id} で更新
- DELETE /api/contacts/{id} で削除
- DB接続は api/db.php の get_pdo() に切り出す
- SQLはPDOのプリペアドステートメントを使う
- nameが空の場合は400エラーを返す
- レスポンスに内部エラー情報を含めない
- このサンプルでは認証は実装しないが、追加・更新・削除APIには本来認証・認可が必要であることをコメントで明記する
```

フロントエンドも生成させる場合は、表示方法の条件も加えます。

```text
フロントエンドでは、ユーザー入力を表示するときに innerHTML を使わず、textContent を使ってください。
```

## AIの限界を知る

### AIが得意なこと

AIは次のようなことを得意としています。

- 決まったパターンのコードを生成する
- ドキュメントや既存のコードをもとに説明する
- 繰り返し作業を素早くこなす
- 複数の選択肢を出す

セクション3から6で作ったPHPのAPIも、AIに生成させると似たようなコードが得られます。

### AIが苦手なこと

一方で、AIには苦手なことがあります。

**1. 文脈の判断**

AIはプロンプトに書かれた情報をもとに出力します。  
「このアプリはどういう目的で使われるか」「どんなユーザーが使うか」といった背景を自分で判断するのは難しいです。

たとえば、「誰でも削除できてよいか」という判断は、アプリの要件次第です。  
AIはその要件を知りません。

**2. 設計の判断**

どの機能をどのファイルに書くか、テーブルをどう設計するかといった設計判断は、AIに任せきりにすると一貫性がなくなることがあります。

セクション8で学んだ「責務分離」のような設計上の考え方を、AIに明示しないと反映されません。

**3. セキュリティの保証**

AIはセキュリティ対策のコードを生成できますが、生成されたコードが必ず安全とは言えません。  
セクション9のチェックポイントを使って、人間が確認する必要があります。

**4. 動作確認の責任**

AIはコードを生成できますが、実際に自分の環境で動くかどうかまでは保証しません。  
curlでAPIを叩く、ブラウザで画面を確認する、SQLiteにデータが保存されているか見る、といった確認は人間が行います。

> **AIはコードを書けますが、責任は取りません。**  
> 動かす判断をするのは、いつも人間です。

## AIにレビューさせる

AIはコード生成だけでなく、既存コードのレビューにも使えます。

たとえば、ここまで作った `api/index.php` をAIに確認させるなら、次のように依頼できます。

```text
以下の api/index.php をレビューしてください。
特に、SQLインジェクション、XSS、認証・認可不足、バリデーション不足、エラー処理の観点で問題を指摘してください。
```

フロントエンドの `app.js` なら、次のように依頼できます。

```text
以下の app.js をレビューしてください。
特に、innerHTMLによるXSSリスク、fetchのエラー処理、フォーム入力値の扱いに問題がないか確認してください。
```

ただし、AIの指摘もそのまま正しいとは限りません。  
セクション8・9で学んだ観点と照らし合わせて、指摘が妥当かどうかを自分で確認します。

## 人間とAIの役割分担

### 何を任せてよいか

AIに任せやすい作業と、人間が判断すべき作業を整理します。

| 作業 | 担当 | 理由 |
|------|------|------|
| CRUDのコード生成 | AI向き | パターンが決まっている |
| SQLの雛形作成 | AI向き | 構文の知識があれば生成しやすい |
| コメント・ドキュメント生成 | AI向き | 既存コードをもとに説明できる |
| セキュリティの確認 | 人間 | コードの意図と文脈を理解する必要がある |
| 動作確認 | 人間 | 自分の環境で本当に動くか確認する必要がある |
| 設計の判断（責務・分離） | 人間 | アプリの要件・背景を知っている |
| バリデーション要件の定義 | 人間 | 何が「正しいデータか」はビジネスルール次第 |
| 認証・認可の設計 | 人間 | 誰に何を許可するかはアプリの要件次第 |

### AIを使うサイクル

実務でAIをうまく使うための流れです。

```text
1. 要件を自分で整理する
   ↓
2. AIにコードを生成させる（具体的なプロンプトで）
   ↓
3. 生成されたコードを読んで理解する
   ↓
4. セキュリティ・設計を確認する
   ↓
5. 動作確認する
   ↓
6. 問題があれば修正する（AIに依頼するか、自分で直す）
```

「生成→確認→修正」のサイクルを回すことが重要です。  
生成されたコードを読まずに使い続けると、問題が起きたときに原因を特定できません。

> **AIは「下書き」を作るのが得意です。**  
> 下書きを最終的な成果物にするのは、読んで判断できる人間の仕事です。

## コードを読む力がなぜ必要か

### 「動く」と「分かる」は違う

このコースの冒頭で、次の問題を紹介しました。

> AIが作ったコードは動く。でも、エラーが出たときに何が起きているか分からない。

ここまでのセクションで、次のことを学んできました。

- HTTPリクエストがどう処理されるか（セクション2・3）
- データがどこに保存されるか（セクション4・5）
- PHPがどうデータベースに接続するか（セクション6）
- フロントエンドとバックエンドがどうつながるか（セクション7）
- コードをどう整理するか（セクション8）
- どんな攻撃があり、なぜ対策が必要か（セクション9）

この知識があると、AIが生成したコードを読んだときに「なぜそう書いているのか」が分かります。  
そして、「ここは安全か」「ここは設計的に問題ないか」を判断できます。

### 知識があるとAIをより活用できる

知識がないままAIを使うと、次のような問題が起きやすくなります。

- 生成されたコードが正しいかどうか判断できない
- エラーが出たとき、何を直せばよいか分からない
- AIに間違ったことを聞いてしまう（前提が間違っていると、答えも間違う）

逆に、基礎知識があると次のことができます。

- 「このコードのSQLはプリペアドステートメントを使っているか」を確認できる
- エラーメッセージから原因を読み取れる
- AIに「この書き方の問題点は何か」と聞いて答えを評価できる

> **AIを使いこなすには、AIに頼らない判断力が必要です。**  
> セクション1から9で学んだ内容は、その判断力の土台です。
